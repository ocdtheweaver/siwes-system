# Technical Documentation
## SIWES Placement, Tracking & Reporting System

This document describes how the system is built, for anyone who needs to
maintain, extend, mark, or defend the code (including for use in Chapter
Four / Appendices of the project report).

---

## 1. Architecture Overview

The system is a server-rendered, multi-page PHP application (no JavaScript
framework, no build step, no REST API layer) following a simple **3-tier
architecture**:

```
┌─────────────────────────────┐
│ Presentation Layer            │  *.php files in /student, /supervisor,
│ (HTML generated by PHP +       │  /admin, /auth, plus includes/header.php,
│  includes/header.php/footer.php│  includes/footer.php, assets/css/style.css
│  + assets/css/style.css)        │
└───────────────┬─────────────┘
                │
┌───────────────▼─────────────┐
│ Application / Logic Layer     │  includes/auth.php (session & access control)
│                                │  includes/functions.php (helpers + matching
│                                │  engine)
└───────────────┬─────────────┘
                │
┌───────────────▼─────────────┐
│ Data Layer                    │  config/db.php (PDO connection)
│                                │  MySQL/MariaDB database "siwes_system"
└─────────────────────────────┘
```

Every page follows the same pattern:

```php
require_once __DIR__ . '/../includes/bootstrap.php';  // config + db + auth + functions
require_role('student');                                // access control guard
// ... page-specific PHP logic (handle POST, run queries) ...
$page_title = '...';
require __DIR__ . '/../includes/header.php';            // shared chrome
// ... page-specific HTML ...
require __DIR__ . '/../includes/footer.php';
```

`includes/bootstrap.php` is the single entry point that loads, in order:
`config/config.php` → `config/db.php` → `includes/auth.php` → `includes/functions.php`.

## 2. Database Schema

Five tables, all in `database/schema.sql`:

| Table              | Purpose                                                              |
|---------------------|------------------------------------------------------------------------|
| `users`              | One row per account. Holds `role` (student/supervisor/admin), credentials. |
| `students`           | Extends a `student`-role user: matric no, department, level, skills, status. |
| `companies`          | Extends a `supervisor`-role user: company profile, required skills, slots. |
| `placements`         | An application/placement linking one student to one company, with status and the match score computed at application time. |
| `logbook_entries`    | Daily activity records, tied to a `placement_id`.                       |
| `reports`            | Weekly/monthly/final reports, tied to a `placement_id`.                 |

**Entity-Relationship summary:**

```
users (1) ──< students (1) ──< placements >── (1) companies >── (1) users
                                    │
                                    ├──< logbook_entries
                                    └──< reports
```

Key design decisions:
- `students` and `companies` are separate tables from `users` (rather than one
  wide table) so that role-specific fields don't pollute every account, and so
  the schema can be extended per role independently.
- `placements` carries a `UNIQUE (student_id, company_id)` constraint so a
  student cannot apply twice to the same company.
- `match_score` is **stored** on the placement row at the moment of
  application (not recalculated later), so the score a student saw when they
  applied is preserved even if either profile changes afterwards - this
  matters for fairness/auditability when a supervisor reviews applicants.
- Status fields use `ENUM` rather than free-text, so invalid states are
  rejected by the database itself, not just by application code.

## 3. Authentication & Access Control

`includes/auth.php`:
- Passwords are hashed with PHP's `password_hash()` (bcrypt by default) and
  verified with `password_verify()`. Plaintext passwords are never stored.
- A PHP session (`$_SESSION`) stores `user_id`, `role`, and `full_name` after
  login.
- `require_login()` and `require_role($role)` are called at the **top** of
  every protected page, before any HTML is output, and redirect via
  `header('Location: ...')` if the check fails. This is why every protected
  page's PHP logic block comes before `includes/header.php` is required.
- There is no separate "permissions" table - access control is role-based
  and enforced per-page (e.g. every file under `/admin/` calls
  `require_role('admin')`), which is sufficient for the three fixed roles
  this system has.

**Role capability matrix:**

| Capability                                   | Student | Supervisor | Admin |
|-----------------------------------------------|:-------:|:----------:|:-----:|
| Edit own profile                              | ✔        | ✔            | -     |
| Browse companies & view match score           | ✔        | -            | -     |
| Apply for a placement                          | ✔        | -            | -     |
| Accept/reject an applicant                     | -        | ✔            | -     |
| Add logbook entries                            | ✔        | -            | -     |
| Comment on a logbook entry                     | -        | ✔            | -     |
| Submit a report                                | ✔        | -            | -     |
| Score / give feedback on a report              | -        | ✔            | -     |
| Mark a placement completed                     | -        | ✔            | ✔ (override) |
| View/manage every student record               | -        | -            | ✔     |
| View/manage every company record               | -        | -            | ✔     |
| Override any placement's status                | -        | -            | ✔     |
| View all reports system-wide                   | -        | -            | ✔     |

## 4. The Recommendation / Matching Engine

Location: `includes/functions.php`, function `compute_match_score()`.

This is a **deliberately transparent, rule-based** algorithm (not machine
learning), so that the basis of every match shown to a student or supervisor
can be explained in one sentence: *"This score is the percentage of the
company's required skills that you have, plus a bonus if your department
relates to their industry."*

```
score = (overlap of student's skills ∩ company's required skills)
        ────────────────────────────────────────────────────────── × 80
                  (count of company's required skills)

        + 20  if student's department text relates to company's industry text
              (else +0)

score is rounded to the nearest whole number, 0-100
```

Both `skills` fields are free-text, comma-separated keyword lists
(e.g. `"php, mysql, html, css"`), split and lower-cased before comparison so
that punctuation/casing differences don't break a match.

This score is used in three places:
1. `student/companies.php` - ranks companies for a student, best match first.
2. `supervisor/dashboard.php` - ranks unplaced students for a company.
3. `supervisor/applicants.php` - shows the stored score for each pending
   applicant so the supervisor can prioritise review.

**Why not a black-box / ML approach?** A handful of training examples (the
number of students and companies any one institution has) is nowhere near
enough data to train a meaningful model, and a rule-based score is something
a non-technical SIWES coordinator can audit and trust. This was a deliberate
scope decision, not an oversight - see Chapter Three, Non-Functional
Requirements.

## 5. Module-by-Module Reference

### `config/`
| File | Purpose |
|---|---|
| `config.php` | Defines `BASE_URL` - the one constant to change when moving between local/hosted environments. |
| `db.php` | Opens the PDO connection (`$pdo`), throwing a clear error if it fails. |

### `includes/`
| File | Purpose |
|---|---|
| `bootstrap.php` | Single include point for config + db + auth + functions. |
| `auth.php` | Session helpers: `login_user()`, `logout_user()`, `is_logged_in()`, `current_role()`, `require_login()`, `require_role()`, `flash()`, `dashboard_url_for_role()`. |
| `functions.php` | `h()` (HTML escaping), `get_student_by_user()`, `get_company_by_user()`, `compute_match_score()`, `match_score_class()`, `status_badge_class()`. |
| `header.php` / `footer.php` | Shared page chrome: nav bar, role-aware sidebar, flash message rendering. |

### `auth/`
| File | Purpose |
|---|---|
| `register.php` | One form, two modes (student/supervisor) toggled by JS; creates a `users` row plus the matching `students`/`companies` row in a transaction. |
| `login.php` | Verifies credentials, starts the session, redirects to the role's dashboard. |
| `logout.php` | Destroys the session. |

### `student/`
`dashboard.php`, `profile.php`, `companies.php` (browse + apply, shows match
score), `logbook.php` (add/view entries, gated on having an accepted
placement), `reports.php` (submit/view reports, same gating).

### `supervisor/`
`dashboard.php` (stats + recommended students), `profile.php` (edit company),
`applicants.php` (accept/reject pending applications, list active students),
`view_student.php` (drill into one placement: comment on logbook entries,
give feedback/score on reports, mark placement completed).

### `admin/`
`dashboard.php` (system-wide stats), `students.php` / `companies.php` /
`placements.php` (search, manually override status, delete), `reports.php`
(read-only system-wide list).

### `database/`
`schema.sql` (DDL only, no fabricated data), `seed.php` (run once in-browser;
creates demo accounts using PHP's own `password_hash()` so the printed
passwords are guaranteed correct - intentionally **not** baked into the SQL
file, since a hash typed directly into SQL cannot be verified without
executing PHP).

## 6. Request Lifecycle Example (Student Applies to a Company)

1. `student/companies.php` loads; for every company it calls
   `compute_match_score()` and sorts best-first.
2. Student clicks "Apply" → browser POSTs `apply_company_id` back to the
   same page.
3. Server checks: is the student already placed? Have they already applied
   to this company? If neither, it inserts a row into `placements` with
   `status = 'pending'` and the computed `match_score`, and (if this is the
   student's first application) flips their `students.status` from
   `unplaced` to `pending`.
4. A flash message confirms success; the page redirects (POST-redirect-GET
   pattern is used everywhere to avoid duplicate submissions on refresh).
5. The company's supervisor sees the application appear in
   `supervisor/applicants.php`, ranked by `match_score`.
6. On **Accept**: a database transaction sets the placement to `accepted`,
   the student to `placed`, and decrements the company's `slots_available`
   by one - all three updates succeed or fail together.
7. On **Reject**: the placement is marked `rejected`; if the student has no
   other pending/accepted application, they're returned to `unplaced` so
   they can apply elsewhere.

## 7. Security Notes

- **SQL injection**: every query that includes user input uses PDO prepared
  statements with bound parameters (`?` placeholders) - no string
  concatenation of user input into SQL anywhere in the codebase.
- **XSS**: every piece of user-controlled data is passed through `h()`
  (an `htmlspecialchars()` wrapper) before being echoed into HTML.
- **Password storage**: bcrypt via `password_hash()`/`password_verify()`,
  never plaintext or reversible encryption.
- **Authorisation**: every page under a role folder calls `require_role()`
  before doing anything else, including before reading `$_GET`/`$_POST`, so
  an unauthenticated or wrong-role request is redirected before any data is
  touched.
- **Cross-record access**: pages that load a specific record by ID (e.g.
  `supervisor/view_student.php?placement_id=5`) always filter by the
  logged-in supervisor's own `company_id` in the SQL `WHERE` clause, so one
  supervisor cannot view or modify another supervisor's students by guessing
  an ID in the URL.
- **CSRF**: not implemented in this version (see Chapter Five / Future Work) -
  acceptable for an academic demonstration system but should be added (e.g.
  a per-session token on state-changing forms) before any real production
  deployment outside a single trusted institution.

## 8. Known Limitations / Deliberate Non-Goals

(Matches Chapter One, Scope of the Project)

- No automatic assignment of students to companies - every placement
  requires a human decision by a supervisor.
- No integration with NYSC, ITF, or any external government database.
- Single-institution design: there is no concept of multiple universities
  sharing one installation.
- No built-in password reset / email notifications (would require an SMTP
  service - out of scope for a local academic deployment).
- No automated test suite (see Chapter Four, Testing and Validation, for the
  manual test cases that were used instead).

## 9. Suggested Future Work

(Expanded in Chapter Five)

1. Add CSRF tokens to all state-changing forms.
2. Add email notifications (e.g. PHPMailer) for application decisions and
   supervisor feedback.
3. Add file upload support for logbooks/reports (currently text-only) and
   for placement-letter documents.
4. Add a password-reset flow.
5. Move the recommendation engine's keyword matching to a normalised
   `skills` table (many-to-many) instead of free-text comma lists, enabling
   features like skill auto-suggest and better partial-match scoring.
